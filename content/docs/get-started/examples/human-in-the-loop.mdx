## What this demonstrates

A single workflow function suspends on a durable promise, exposes the promise ID to the outside world, and resumes the moment that promise resolves. The same mechanic doubles as deduplication (re-invoking the same workflow ID reconnects to the in-flight execution) and survives worker crashes (kill the worker mid-block, the workflow recovers on another).

## The problem

Real business processes routinely depend on human input — a manager approves a refund, a reviewer accepts a generated summary, a customer confirms an address. The wait can be seconds or weeks, and the answer can come from email, a webhook, a UI, or a CLI.

Solving this without a runtime that understands suspension means breaking the process apart by hand: a queue to park the work, a job to wake it up, a database row to track which step you're on, a polling loop somewhere. Every shop reinvents the same coordination layer, and every implementation has its own subtle bugs around retries, races, and recovery.

## Resonate's solution

Yield a Resonate Promise from inside a workflow and execution suspends until that promise resolves. The promise has a stable ID; anything that knows the ID — another process, a CLI, an HTTP handler, a button click — can resolve it. There's no scheduler, no parking queue, no external state machine.

Because the suspension lives inside a durable promise, it survives process crashes. A worker that's blocking on a human can be killed and the workflow continues on whichever worker picks the task back up.

## Code walkthrough

The example has two processes: a **worker** that runs the workflow, and a **gateway** that starts workflows over HTTP and resolves their blocking promises.

### The blocking workflow

The workflow creates a promise, makes its ID visible (here via an `sendEmail` step that prints a callback URL), then yields the promise. Execution pauses until something resolves it.

```typescript title="worker.ts"
function* foo(ctx: Context, workflowId: string): Generator<any, void, any> {
  // Create a durable promise. Its ID is derived from the workflow ID.
  const blockingPromise = yield* ctx.promise();

  // Make the promise ID reachable from outside — email, webhook, log, etc.
  yield* ctx.run(sendEmail, blockingPromise.id);
  console.log(`blocked, waiting on human interaction`);

  // Suspend until the promise resolves. Survives crashes.
  yield* blockingPromise;
  console.log(`unblocked, promise resolved`);
}

resonate.register("foo-workflow", foo);
```

def foo(ctx, workflow_id):

### Resolving the promise

The gateway exposes two endpoints — one to start a workflow, one to resolve the blocking promise. Resolving the promise unblocks every workflow waiting on that ID, anywhere in the cluster.

```typescript title="gateway.ts"
app.post("/start-workflow", async (req, res) => {
  const { workflow_id } = req.body ?? {};
  // Same workflow_id reconnects to a PENDING execution rather than starting a new one.
  const result = await resonate.rpc(
    workflow_id,
    "foo-workflow",
    workflow_id,
    resonate.options({ target: "poll://any@workers" }),
  );
  res.status(200).json({ message: result });
});

app.get("/resolve/:promiseId", async (req, res) => {
  await resonate.promises.settle(req.params.promiseId, "resolved");
  res.status(200).json({ message: "promise resolved" });
});
```

</TabItem>

<TabItem value="python">

```python title="gateway.py"
@app.route("/start-workflow", methods=["POST"])
def start_workflow():
    data = request.get_json() or {}
    workflow_id = data["workflow_id"]
    # Same workflow_id reconnects to a PENDING execution rather than starting a new one.
    handle = resonate.options(target="poll://worker").rpc(
        workflow_id, "foo", workflow_id
    )
    if handle.done():
        return jsonify({"message": handle.result()})
    return jsonify({"message": f"workflow {workflow_id} started"}), 200

@app.route("/resolve/<promise_id>", methods=["GET"])
def resolve(promise_id):
    resonate.promises.resolve(id=promise_id)
    return jsonify({"message": "promise resolved"})
```

## Run it locally

You'll need the Resonate Server, the worker, and the gateway running. Pick the language tab.

<Tabs
    groupId="preferred-language"
    defaultValue="typescript"
    values={[
        {label: 'TypeScript', value: 'typescript'},

<TabItem value="typescript">

Clone the repo and install dependencies (uses [Bun](https://bun.sh/)):

```shell
git clone https://github.com/resonatehq-examples/example-human-in-the-loop-ts
cd example-human-in-the-loop-ts
bun install
```

In separate terminals, start the Resonate Server, the worker, and the gateway:

```shell title="Terminal 1 — Resonate Server"
brew install resonatehq/tap/resonate
resonate dev
```

```shell title="Terminal 2 — gateway"
bun run gateway.ts
```

```shell title="Terminal 3 — worker"
bun run worker.ts
```

Start a workflow:

```shell
curl -X POST http://localhost:5001/start-workflow \
  -H "Content-Type: application/json" \
  -d '{"workflow_id": "hitl-001"}'
```

The worker prints a callback URL. Hit it (or the matching `/resolve/:promiseId` endpoint) to unblock the workflow.

</TabItem>

## Try the recovery story

The same setup demonstrates Resonate's load balancing and crash recovery for free. Run multiple workers in parallel:

```shell
bun run worker.ts &
bun run worker.ts &
bun run worker.ts &
```

Start several workflows with different IDs and watch them spread across the workers. Then kill one of the workers while a workflow is blocked — Resonate reassigns it and the workflow resumes on a survivor.

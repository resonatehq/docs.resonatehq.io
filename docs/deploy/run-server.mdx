This page will help you install and run the Resonate Server.

First install the Server on your machine.

## Install with Homebrew

You can install the latest Resonate Server version with Homebrew via:

```shell
brew install resonatehq/tap/resonate
```

All available releases and associated release artifacts are available on [Github](https://github.com/resonatehq/resonate/releases).

Once installed, you can start the server with an in-memory store by running:

```shell title="Run using an in-memory store"
resonate dev
```

Or, you can run the server with the default SQLite store by running:

```shell title="Run using the default SQLite store"
resonate serve
```

You will see log output like the following:

```shell
resonate: Server listening port=8001
resonate: Metrics server listening port=9090
```

The output indicates that the server has an HTTP API available at port 8001 and a metrics endpoint at port 9090.

These are the default ports and can be changed via CLI flags.

## Run with Docker

There is no official Docker image yet.
To run the server in Docker, wrap the release tarball in a Dockerfile.

:::caution GLIBC requirement
The Resonate binary requires **glibc ≥ 2.38**. Use `debian:trixie-slim` or `ubuntu:24.04` as the base image. Older images like `debian:bookworm` or `ubuntu:22.04` will fail.
:::

```dockerfile title="Dockerfile"
ARG RESONATE_VERSION=v0.9.3

FROM debian:trixie-slim AS fetch
ARG RESONATE_VERSION
ARG TARGETARCH=x86_64
RUN apt-get update \
 && apt-get install -y --no-install-recommends curl ca-certificates tar \
 && rm -rf /var/lib/apt/lists/*
WORKDIR /src
RUN TARBALL="resonate_linux_${TARGETARCH}.tar.gz" \
 && curl -fsSL -O "https://github.com/resonatehq/resonate/releases/download/${RESONATE_VERSION}/${TARBALL}" \
 && tar -xzf "${TARBALL}" \
 && test -x ./resonate

FROM debian:trixie-slim AS runtime
RUN apt-get update \
 && apt-get install -y --no-install-recommends ca-certificates \
 && rm -rf /var/lib/apt/lists/* \
 && groupadd -r resonate && useradd -r -g resonate -d /app -s /usr/sbin/nologin resonate
WORKDIR /app
COPY --from=fetch /src/resonate /usr/local/bin/resonate
USER resonate
EXPOSE 8001

CMD ["resonate", "serve", \
     "--server-host", "0.0.0.0", \
     "--server-port", "8001", \
     "--level", "info"]
```

Build and run:

```shell
docker build -t resonate-server .
docker run -p 8001:8001 resonate-server
```

To use PostgreSQL, pass the storage flags:

```shell
docker run -p 8001:8001 resonate-server \
  resonate serve \
  --server-host 0.0.0.0 \
  --server-port 8001 \
  --storage-type postgres \
  --storage-postgres-url "postgres://user:pass@host:5432/resonate"
```

## Deploy to GCP Cloud Run

To deploy an instance of the Resonate Server to Google Cloud Run, start by deploying a new service.
Select "Deploy container" from the Cloud Run dashboard.

![Deploy container](/img/cloud-run-dashboard.png)

On the configuration screen, select "Deploy one revision from an existing container image".

Build and push the Dockerfile above to your container registry, then enter the image URL.

![Docker image URL](/img/docker-image-url.png)

Set the "Container port" to "8001", which is the default port for the Resonate Server's HTTP API.

And ensure that there is just 1 instance of the server running.
Use "Manual scaling" and set "Number of instances" to "1".

![Cloud Run instance settings](/img/cloud-run-service-scaling.png)

Set the following environment variables:

| Variable | Value |
|----------|-------|
| `RESONATE_SERVER_HOST` | `0.0.0.0` |
| `RESONATE_SERVER_PORT` | `8001` |
| `DATABASE_URL` | Your Postgres connection string |

You should be able to customize the rest of the settings as needed for your deployment.

## Build from source

If you don't have Homebrew, you can build from source using Rust.
Run the following commands to download the repository and build the server:

```shell
git clone https://github.com/resonatehq/resonate
cd resonate
cargo build --release
```

After it is built, run the binary directly:

```shell
./target/release/resonate serve
```

## CLI flags reference

The server is configured via CLI flags.
Run `resonate serve --help` for a full list.

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--server-host` | string | `0.0.0.0` | HTTP server bind address |
| `--server-port` | int | `8001` | HTTP server port |
| `--server-url` | string | — | Server's externally-reachable URL (required for distributed deployments where workers call back to the server) |
| `--level` | string | `info` | Log level: `debug`, `info`, `warn`, `error` |
| `--tasks-retry-timeout` | int (ms) | `30000` | Suspend/wake retry interval. Lower values (e.g. `500`) significantly reduce latency for chained or recursive workflows |
| `--storage-type` | string | `sqlite` | Storage backend: `sqlite` or `postgres` |
| `--storage-sqlite-path` | string | `/tmp/resonate.db` | Path to SQLite database file |
| `--storage-postgres-url` | string | — | Postgres connection string (e.g. `postgres://user:pass@host:5432/resonate`). Required when `--storage-type postgres` |
| `--storage-postgres-pool-size` | int | `10` | Postgres connection pool size |
| `--auth-publickey` | string | — | Path to JWT public key (PEM format) for authentication |
| `--transports-gcps-project` | string | — | GCP Pub/Sub project ID for message transport |

:::tip tasks-retry-timeout
The default `--tasks-retry-timeout` of 30 seconds adds significant latency to chained workflows (sagas, recursive fan-out).
For most deployments, set this to `500` (milliseconds) to keep suspend/wake cycles fast.
:::

## Run with PostgreSQL

For production deployments, use PostgreSQL as the storage backend:

```shell
resonate serve \
  --storage-type postgres \
  --storage-postgres-url "postgres://user:pass@localhost:5432/resonate" \
  --storage-postgres-pool-size 20
```

Migrations are applied automatically on first startup.

See the [Availability](/deploy/availability) guide for high-availability PostgreSQL configuration.

## Run on Kubernetes

The recommended Kubernetes deployment strategy is a single-replica deployment:

```yaml
---
apiVersion: v1
kind: Service
metadata:
  name: resonate
spec:
  selector:
    app: resonate
  ports:
    - port: 8001
      name: api
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: resonate
spec:
  replicas: 1
  selector:
    matchLabels:
      app: resonate
  template:
    metadata:
      labels:
        app: resonate
    spec:
      containers:
        - name: resonate
          image: your-registry/resonate-server:v0.9.3
          command: ["resonate", "serve"]
          args:
            - "--server-host=0.0.0.0"
            - "--server-port=8001"
            - "--storage-type=postgres"
            - "--storage-postgres-url=$(DATABASE_URL)"
            - "--tasks-retry-timeout=500"
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: resonate-secrets
                  key: database-url
          ports:
            - name: api
              containerPort: 8001
```

Create a file named `resonate-k8s.yaml` and apply the manifest:

```console
kubectl apply -f resonate-k8s.yaml
```

To remove Resonate from your cluster:

```console
kubectl delete -f resonate-k8s.yaml
```

:::info Health checks
The server does not expose a `/healthz` endpoint.
A health probe can use `GET /` which returns `405 Method Not Allowed` — this confirms the server is alive and accepting connections.
:::

## Ports

The server binds two ports:

| Port | Purpose |
|------|---------|
| `:8001` | HTTP API (all SDK traffic, SSE polling, and task dispatch) |
| `:9090` | Prometheus metrics |

:::caution PaaS port routing
On platforms like Railway, Render, or Fly.io, explicitly set `PORT=8001` or `RESONATE_SERVER_PORT=8001`.
Without this, the platform may auto-detect the metrics port (`:9090`) as the primary port and route all traffic there, resulting in 502 errors.
:::
